Architectural Synthesis of Cloud-Native Attack Vectors: Threat Intelligence, Telemetry, and Detection EngineeringExecutive SummaryThe rapid maturation of container orchestration platforms has been paralleled by an unprecedented evolution in adversary tradecraft. Between the years 2024 and 2026, the cloud-native threat landscape experienced a profound paradigm shift, moving decisively away from simple, opportunistic cryptojacking toward highly targeted, sophisticated operations orchestrated by Advanced Persistent Threats (APTs) and elite cybercriminal syndicates. Recent intelligence indicates that the financial implications of these intrusions have reached critical levels; operations such as "LLMjacking" have demonstrated the capability to inflict financial damages exceeding $100,000 per day by hijacking cloud resources allocated to computationally expensive Large Language Models (LLMs). The average cost of a data breach within cloud environments reached $5.17 million in 2024, significantly outpacing the costs associated with traditional on-premises compromises.Defending modern Kubernetes (K8s) environments requires moving beyond static vulnerability management and legacy perimeter defenses. Attackers continually leverage ephemeral execution contexts, sophisticated container escape techniques, and native API abuses to traverse the kill chain seamlessly without triggering signature-based alarms. Elite threat actors, including the operators behind SCARLETEEL, Kinsing, and nation-state groups like the Democratic People's Republic of Korea (DPRK) nexus FAMOUS CHOLLIMA, have demonstrated alarming proficiency in subverting traditional security controls, operating entirely within system memory, and utilizing native shell built-ins to evade endpoint detection. Concurrently, the weaponization of Artificial Intelligence has acted as a force multiplier, compressing the attack lifecycle from initial access to full administrative impact to mere minutes.This comprehensive research report synthesizes real-world Cyber Threat Intelligence (CTI) from recent post-mortem analyses into a cohesive, actionable framework engineered for Security Operations Center (SOC) Analysts, Detection Engineers, and Cloud Security Architects. It meticulously deconstructs the technical mechanics of prominent container attack vectors, mapping them to the MITRE ATT&CK framework. Furthermore, it details the architectural nuances of telemetry collection using extended Berkeley Packet Filter (eBPF) and Kubernetes Audit Logs across diverse environments, including standard clusters, Amazon EKS Auto Mode, and serverless compute models like AWS Fargate. The analysis provides concrete "Parsers as Code" and advanced detection rules tailored for Next-Generation Security Information and Event Management (SIEM) platforms. Finally, the report translates these defensive telemetry requirements into preventative architectural hardening guidelines, focusing on admission control and network policies designed to disrupt the attacker kill chain deterministically prior to execution.Pillar 1: Cyber Threat Intelligence (CTI) & Technical Kill ChainsUnderstanding the precise mechanical execution of cloud-native intrusions is the foundational requirement for engineering effective detection logic. Over the past 36 months, threat intelligence telemetry has identified a sharp increase in attacks targeting misconfigured containerized applications, exploiting novel vulnerabilities at the application layer, and abusing native cloud services. Analyzing these real-world incidents reveals a consistent, multi-stage attack lifecycle tailored specifically for the distributed nature of Kubernetes.Initial Access: Public-Facing Applications and Pre-Authentication RCEAdversaries frequently achieve initial access by exploiting public-facing applications hosted within Kubernetes clusters, relying heavily on the MITRE ATT&CK technique T1190 (Exploit Public-Facing Application). A defining example from the 2025–2026 timeframe is the exploitation of React2Shell (CVE-2025-55182), a critical insecure deserialization vulnerability residing within the React Server Components "Flight" protocol.The Flight protocol vulnerability allows an unauthenticated attacker to inject malicious, serialized data during the initialization phase. The exploit leverages JavaScript duck-typing and dynamic code execution to create a self-reference loop, effectively tricking the JavaScript engine into calling attacker-controlled code. Within 30 hours of the vulnerability's patch release on December 3, 2025, opportunistic attackers successfully weaponized the flaw to load the Node.js child_process module, granting arbitrary remote code execution (RCE) on the underlying pods. By converting unauthenticated internet access into immediate command execution inside a workload, threat actors bypassed traditional network perimeter controls and gained a crucial foothold within the cluster's logical network namespace.Similarly, the SCARLETEEL 2.0 operation leveraged compromised containerized workloads exposed directly to the public internet. Specifically, the attackers targeted JupyterLab notebook containers operational within a self-managed Kubernetes cluster hosted in an AWS environment. These data science environments, often deployed with excessive IAM permissions and lacking stringent authentication protocols, provided an ideal environment for the attackers to initiate the next phase of their operation without triggering traditional ingress alarms.Execution and Lateral Movement: Native Tooling and API AbuseOnce initial access is secured inside a containerized workload, attackers must navigate the cluster to discover high-value assets. Rather than introducing custom, compiled malware that might trigger static anti-virus signatures or runtime behavioral blocks, advanced actors increasingly "live off the land" by downloading legitimate cloud administration tools or utilizing specialized Kubernetes penetration testing utilities.In the SCARLETEEL 2.0 campaign, following the initial breach of the JupyterLab pod, the attackers deployed the Peirates utility—an open-source Kubernetes penetration testing tool designed to escalate privileges and pivot through a cluster. Utilizing the default Service Account token, which is automatically mounted within nearly all Kubernetes pods at /var/run/secrets/kubernetes.io/serviceaccount/token unless explicitly disabled, the tool rapidly queried the Kubernetes API server. The actors issued specific verbs (get, list) against critical resources (secrets, pods, namespaces) to map the internal cluster topology and harvest sensitive credentials.To move laterally from the isolated container into the broader cloud infrastructure, the attackers targeted the cloud provider's Instance Metadata Service (IMDS). By issuing HTTP GET requests to the "magic" IP address 169.254.169.254, the attackers retrieved highly sensitive AWS credential data associated with the underlying EC2 worker node's IAM role. With these credentials in hand, the attackers subsequently modified the HOME environment variable within the container to /root to ensure subsequent scripts executed with elevated local privileges. They then installed the Pacu exploitation framework and the AWS CLI, utilizing the stolen IMDS credentials to enumerate and compromise the wider AWS account, bypassing the container boundary entirely.Privilege Escalation: Container Escapes via File Descriptor LeaksThe most critical phase of a Kubernetes intrusion is the container escape, wherein an attacker breaks the logical isolation boundary of the container namespace to achieve root-level execution on the underlying host node. Once the host node is compromised, the attacker can access the kubelet credentials, compromise other tenant containers on the same node, and potentially take over the entire cluster.The "Leaky Vessels" vulnerability suite, specifically CVE-2024-21626 affecting the runc container runtime, represents a watershed moment for privilege escalation techniques utilized by malware families such as Kinsing. The vulnerability relies on an internal file descriptor leak within runc versions up to and including v1.1.11. When a container is spawned, the leaked file descriptor provides a reference back to the host operating system's filesystem.An attacker can exploit this in a Kubernetes environment by manipulating the WORKDIR directive of a malicious container image or during an API exec command. By setting the working directory to the leaked file descriptor via the procfs virtual filesystem (e.g., WORKDIR /proc/self/fd/9), the initialized process conceptually starts with its root directory mapped to a directory on the host instance, such as /sys/fs/cgroup. The attacker subsequently executes a path traversal attack (cd../../../../../) to break out of the containerized root, granting the ability to read or overwrite critical host binaries and secrets, such as /etc/shadow or the kubelet configuration files.Defense Evasion: Memory-Only Execution and Log TamperingTo maintain persistence and evade detection during these operations, cloud-native adversaries employ sophisticated defense evasion techniques. The SCARLETEEL operators demonstrated a nuanced understanding of Linux system logging by persistently executing the rm command against /var/log/syslog and other localized logging sources. This action effectively blinded agent-based log forwarders before the events could be shipped to a centralized SIEM. They also utilized commands such as history -cw to clear the session's command history and clear to wipe the terminal interface.Furthermore, data exfiltration was conducted using Bash shell built-ins rather than standard utilities like curl or wget. Standard security protocols often flag the execution of network utilities by web server processes as anomalous. By utilizing shell built-ins to open TCP sockets (e.g., directing output to /dev/tcp), the attackers successfully bypassed static behavioral detections. The exfiltrated data, including proprietary source code and AWS .git-credentials files, was Base64 encoded, stored in memory variables (SEND_B64_DATA), and transmitted directly to Russian infrastructure compatible with the S3 protocol (hb.bizmrg.com) or to temporary pastebin-like endpoints (temp.sh).Summary of Advanced Threat Actor Campaigns (2024-2026)To synthesize the evolving threat landscape, the following table outlines the prominent campaigns, their primary objectives, and the specific MITRE ATT&CK techniques utilized.Threat Actor / CampaignTarget InfrastructurePrimary ObjectiveKey TTPs & ExploitsSCARLETEEL 2.0AWS EKS, JupyterLabProprietary Code Theft, CryptominingT1190 (Public-Facing App), T1528 (Steal Token), IMDS Abuse (169.254.169.254), Shell Built-in Exfiltration, Peirates tool.FAMOUS CHOLLIMA (DPRK)Multi-Cloud, KubernetesExtortion, RansomwareGenAI-accelerated reconnaissance, Insider threat personas, Malware-free intrusion (81% of operations).Unknown (React2Shell)React Server ComponentsRemote Code ExecutionCVE-2025-55182 (Flight Protocol Deserialization), Node.js child_process execution.Kinsing / TeamTNTDocker, K8s, FargateCryptojacking, BotnetCVE-2024-21626 (runc escape), DaemonSet abuse, XMRig deployment, SSH-Snake worm propagation.LLMjacking OperatorsCloud AI InfrastructureResource HijackingAutomated credential theft targeting Anthropic Claude 3 / OpenAI GPT hosting accounts, costing victims up to $100,000/day.Pillar 2: Telemetry Collection & Visibility NuancesDetecting the highly dynamic and sophisticated attack chains detailed above requires a fundamental shift in enterprise telemetry architecture. Traditional Endpoint Detection and Response (EDR) solutions, which rely heavily on static file scanning, registry monitoring, and periodic process polling, are insufficient in highly ephemeral container environments where pods may only exist for seconds. Comprehensive visibility necessitates a dual-layered approach: control plane monitoring via Kubernetes Audit Logs, and real-time data plane monitoring via kernel-level eBPF tracing.The Control Plane: Architecting Kubernetes Audit LoggingKubernetes Audit Logs provide a security-relevant, chronological record of all API interactions within a cluster. They answer the critical questions of who initiated an action, what resource was targeted, from where the request originated, and what the final outcome was. Because attackers rely heavily on the API server to perform reconnaissance (e.g., listing secrets), deploy privileged pods, and execute remote commands within running workloads (via kubectl exec), audit logs act as the absolute primary detection surface for initial cluster compromise and internal lateral movement.Audit records undergo a specific lifecycle within the kube-apiserver component and can be recorded at distinct stages:RequestReceived: The event is generated as soon as the audit handler receives the request, prior to delegation down the handler chain.ResponseStarted: Generated once response headers are sent, but before the body is sent (primarily for long-running requests like watch).ResponseComplete: The response body has been fully transmitted.A major architectural challenge for SOC and DevOps teams is balancing granular visibility with performance degradation and storage costs. Logging every single cluster event at the RequestResponse level can drastically increase the memory consumption of the API server and flood SIEM infrastructure with millions of benign operational events per day.To optimize threat detection while managing ingestion costs, the audit-policy.yaml must be explicitly engineered to capture the exact verbs and resources associated with adversarial techniques.YAMLapiVersion: audit.k8s.io/v1
kind: Policy
omitStages:
  - "RequestReceived" # Reduce noise by ignoring initial requests without final outcomes
rules:
  # High-Fidelity Threat Detection: Exec, Attach, and PortForward
  - level: RequestResponse
    resources:
    - group: "" # Core API group
      resources: ["pods/exec", "pods/attach", "pods/portforward"]
  
  # Privilege Escalation Detection: Pod Creation and Modifications
  - level: Request
    verbs: ["create", "update", "patch", "delete"]
    resources:
    - group: ""
      resources: ["pods", "serviceaccounts", "bindings"]
    - group: "rbac.authorization.k8s.io"
      resources: ["roles", "rolebindings", "clusterroles", "clusterrolebindings"]

  # Security & Secret Management: Log metadata only to prevent secret leakage
  - level: Metadata
    verbs: ["create", "update", "get", "list", "watch"]
    resources:
    - group: ""
      resources: ["secrets", "configmaps"]

  # Default Catch-All for remaining actions
  - level: Metadata
    omitStages:
      - "RequestReceived"
Architectural Context & Risk Analysis: By configuring pods/exec at the RequestResponse level, defenders capture the exact command-line arguments passed to the container shell by an attacker. However, the policy deliberately restricts the logging of secrets and configmaps strictly to the Metadata level. Logging secrets at the RequestResponse level results in plaintext credential exposure within the SIEM environment—a severe misconfiguration that threat actors actively hunt for during the post-exploitation phase. Furthermore, minimizing wildcard use in Roles and ClusterRoles is explicitly recommended by the CIS GKE Benchmark v1.5.0 to adhere to the principle of least privilege.The Data Plane: System Call Visibility via eBPFWhile audit logs flawlessly cover the control plane, they remain entirely blind to the execution context inside the container. To observe file descriptor manipulation (CVE-2024-21626), shell built-in exfiltration techniques (SCARLETEEL), or memory-only cryptominers masking themselves as legitimate systemd processes, defenders require real-time visibility into the Linux kernel.Extended Berkeley Packet Filter (eBPF) has definitively emerged as the foundational technology for cloud-native runtime security. eBPF allows for the execution of highly efficient, sandboxed programs directly within the Linux kernel space without requiring complex kernel module modifications or system reboots. By safely hooking into Kprobes (dynamic kernel function tracing), Tracepoints (static kernel hooks), and networking queues (XDP), eBPF captures highly granular telemetry—such as execve (process execution), openat (file access), and TCP connection states—with minimal performance overhead, typically benchmarking at 1-2.5% CPU utilization.The architectural implementation of eBPF varies significantly among the dominant open-source Cloud Native Computing Foundation (CNCF) security projects:eBPF Security ToolPrimary Design PhilosophyeBPF Interaction ModelMitigation CapabilityFalcoDetection & AlertingAsynchronous. Detects after the fact.Tetragon (Cilium)Real-time EnforcementSynchronous. Uses advanced eBPF filters directly within kernel-space to evaluate conditions.Active. Can issue bpf_send_signal to drop malicious syscalls before execution occurs.Tracee (Aqua)Deep Forensic TracingModerate to high overhead deep tracing designed for intensive forensic data collection.Passive. Best utilized for incident response and deep behavioral analysis.Tetragon’s architecture represents a significant leap forward for SOC teams requiring active mitigation. If a policy prohibits writing to /etc/shadow, Tetragon's eBPF program evaluates the condition within the kernel itself and drops the syscall synchronously before the action succeeds, effectively preventing container escapes in real-time. Conversely, Falco remains the industry standard for vast rule libraries and broad detection coverage, offering seamless integration with tools like Fluent Bit and AWS Firelens for robust log routing to SIEM platforms.Architectural Nuances: Managed Control Planes vs. Serverless ContainersDeploying eBPF sensors introduces significant architectural complexities depending heavily on the underlying cluster topology chosen by the organization.In standard Amazon EKS or Google Kubernetes Engine (GKE) environments, eBPF security agents are deployed as highly privileged DaemonSets. A DaemonSet ensures that exactly one instance of the security agent runs on every underlying worker node. Because the agent requires hostPID, hostNetwork, and privileged capabilities to load eBPF byte-code into the shared host kernel and monitor all co-located pods, this model necessitates full access to the underlying virtual machine.However, the cloud industry is rapidly transitioning toward automated, serverless provisioning models, such as Amazon EKS Auto Mode and AWS Fargate, which drastically alter telemetry collection strategies.Amazon EKS Auto Mode: In this model, AWS completely automates the management of the data plane, utilizing Karpenter for auto-scaling compute and managing node rotation. However, the underlying EC2 instances remain distinct execution environments. Therefore, deploying Falco or Tetragon via a DaemonSet is still fully supported. This allows organizations to maintain standard, node-level eBPF deployment topologies while simultaneously reducing node management overhead.AWS Fargate (Serverless): Fargate completely abstracts the concept of the worker node from the user. Consequently, DaemonSets are entirely unsupported, and containers absolutely cannot be run in privileged mode or with hostNetwork access. Attempting to deploy kernel-level eBPF agents via DaemonSet is architecturally impossible in a pure Fargate environment. Security in these serverless paradigms must rely heavily on application-level sidecar injection—which introduces high resource duplication and complexity—alongside AWS-native network telemetry and strict API Audit Log monitoring.Pillar 3: Detection Engineering & "Parsers as Code"Transforming raw, unstructured cloud telemetry into high-fidelity, actionable alerts requires a robust Detection Engineering pipeline. Analysts face the daunting task of normalizing varying log formats from different Cloud Service Providers (CSPs) into a consistent schema, and subsequently developing complex correlation logic to track adversary lateral movement across the container boundary.Log Normalization and the Unified Data Model (UDM)Kubernetes audit logs are highly nested, verbose JSON structures that do not cleanly align with traditional SIEM fields out of the box. In advanced platforms like Google Security Operations (Chronicle SIEM), raw data must be mapped to the Unified Data Model (UDM) through custom "Parsers as Code" to enable efficient search, dashboarding, and threat hunting capabilities.The UDM schema relies on standard entities such as principal (the actor initiating the action), target (the impacted resource), and metadata.event_type (the systemic categorization of the action). Vendor-specific modifications compound this challenge; for example, Google Kubernetes Engine (GKE) heavily customizes native audit logs, introducing specific payloads and altering the original Kubernetes fields. If a log parser does not explicitly extract the Kubernetes user.username and map it to the UDM field principal.user.userid, detection rules relying on User Behavior Analytics (UBA) or identity tracking will catastrophically fail.Below is an architectural mapping table demonstrating how critical Kubernetes Audit fields must be translated into the UDM schema to preserve analytical value :K8s Audit Log FieldUDM Schema FieldAnalytical Purpose for Detection Engineeringuser.usernameprincipal.user.useridTracks the specific Service Account or human user initiating the API request.sourceIPsprincipal.ipIdentifies the origin of the API request (e.g., an external attacker IP or an internal pod IP).objectRef.resourcetarget.resource.nameIdentifies the impacted Kubernetes resource (e.g., pods, secrets).objectRef.namespacetarget.resource.namespaceIsolates the logical environment (e.g., kube-system, default).verb (e.g., create)metadata.event_typeNormalizes to constants like USER_RESOURCE_CREATION or USER_RESOURCE_ACCESS.requestURIsecurity_result.detection_fields[command_line]Captures the exact command passed during a kubectl exec session for regex matching.To execute this normalization, Detection Engineers must write custom parsers.   // Verify the log is a valid Kubernetes Audit Event
  if [kind] == "Event" and [apiVersion] =~ "audit.k8s.io" {
    
    // Categorize the event type based on the requested subresource
    if[subresource] == "exec" {
      mutate {
        replace => {
          "metadata.event_type" => "USER_RESOURCE_ACCESS"
          "metadata.description" => "Kubernetes Pod Execution"
        }
      }
    } else if [verb] == "create" and[resource] == "pods" {
      mutate {
        replace => { "metadata.event_type" => "USER_RESOURCE_CREATION" }
      }
    }

    // Map the Principal (Actor) attributes
    mutate {
      rename => {
        "[user][username]" => "principal.user.userid"
        "[sourceIPs]" => "principal.ip"
        "[userAgent]" => "network.http.user_agent"
      }
    }

    // Map the Target (Namespace and Pod) attributes
    mutate {
      rename => {
        "[namespace]" => "target.resource.namespace"
        "[name]" => "target.resource.name"
      }
    }

    // Capture the exact command executed from the URI parameters for deep inspection
    if {
      grok {
        match => { "requestURI" => ".*command=(?<executed_command>[^&]+).*" }
      }
      mutate {
        replace => { "security_result.detection_fields[command_line]" => "%{executed_command}" }
      }
    }
  }
}
Architectural Context: Mapping the exact requestURI query parameters into a custom security_result.detection_fields allows Detection Engineers to write high-fidelity rules searching for specific Base64-encoded strings, shell built-in executions, or offensive tools like Peirates and XMRig, bridging the gap between the raw API request and the SIEM's behavioral analytical engine.YARA-L 2.0: Multi-Stage Telemetry CorrelationDetecting advanced threats like SCARLETEEL requires correlating events across completely different data streams. A single kubectl exec command might be benign developer troubleshooting. A container establishing an outbound connection might be a legitimate application fetching an update. However, an unauthenticated user establishing a remote execution session into a pod, followed immediately by that exact pod launching an anomalous outbound network connection to an unknown IP, represents a critical security incident.Using Google SecOps' YARA-L 2.0 query language, analysts can leverage multi-stage outer joins to correlate Kubernetes control plane audit logs directly with data plane eBPF network flows.Code snippetrule K8s_Pod_Exec_Followed_By_Suspicious_Outbound_Connection {
  meta:
    author = "Detection Engineering Team"
    description = "Detects a kubectl exec session into a pod followed rapidly by a network connection to a non-standard external IP, indicating potential reverse shell or data exfiltration."
    severity = "CRITICAL"
    mitre_attack_tactic = "Execution, Exfiltration"
    mitre_attack_technique = "T1609, T1041"

  events:
    // Stage 1: Kubernetes API Audit Log capturing the Exec command
    $k8s_exec.metadata.event_type = "USER_RESOURCE_ACCESS"
    $k8s_exec.metadata.description = "Kubernetes Pod Execution"
    $k8s_exec.principal.ip = $attacker_ip
    $k8s_exec.target.resource.name = $target_pod
    $k8s_exec.target.resource.namespace = $target_namespace

    // Stage 2: eBPF (e.g., Tetragon/Cilium) Network Event capturing the outbound connection
    $ebpf_net.metadata.event_type = "NETWORK_CONNECTION"
    $ebpf_net.principal.resource.name = $target_pod
    $ebpf_net.principal.resource.namespace = $target_namespace
    $ebpf_net.target.ip = $destination_ip

    // Exclude trusted internal network ranges and cloud provider IPs to reduce noise
    not net.ip_in_range_cidr($destination_ip, "10.0.0.0/8")
    not net.ip_in_range_cidr($destination_ip, "172.16.0.0/12")

  match:
    // Correlate the two distinct events on the specific Pod name and Namespace
    $target_pod, $target_namespace over 5m

  condition:
    // Trigger if both events occur, and the network event happens AFTER the exec session
    $k8s_exec and $ebpf_net and $k8s_exec.metadata.event_timestamp < $ebpf_net.metadata.event_timestamp
}
Architectural Context: This rule explicitly maps the logical concept of a Kubernetes Pod ($target_pod) across two discrete telemetry layers (Control Plane and Data Plane). It effectively negates the false positives generated by automated health checks or routine operations by explicitly tying the network anomaly to a preceding interactive user execution session.Sigma: Open-Source Correlation RulesFor vendor-agnostic deployments, Detection Engineers heavily utilize Sigma rules. Architectural Context: By correlating the administrative action of creating a privileged workload with the runtime behavior of accessing restricted host files, the Sigma rule filters out legitimate cluster administration tasks (like deploying a required daemon) and isolates genuine malicious breakout attempts.Pillar 4: Real-World Defensive Guidance & HardeningHigh-fidelity detection logic is essential for incident response, but it inherently remains a reactive measure. Mature cloud-native security postures focus heavily on architectural mitigations that break the adversary's kill chain deterministically, rendering entire classes of vulnerabilities unexploitable regardless of the specific malware deployed.Breaking Initial Access and Escalation: Admission ControllersTo mitigate container escapes such as the runc CVE-2024-21626 exploit utilized by Kinsing, clusters must enforce strict logical boundaries prior to workload instantiation. Attackers rely entirely on misconfigured securityContext settings, excessive Linux capabilities, or the ability to mount host file systems to execute the breakout path.Admission controllers, such as Kyverno or OPA Gatekeeper, act as the ultimate gatekeepers. They intercept requests to the Kubernetes API server prior to object persistence, evaluating the workload specification against a library of security policies. Using Common Expression Language (CEL), Kyverno can enforce Pod Security Standards (PSS) dynamically.Below is a Kyverno ClusterPolicy designed to explicitly deny hostPath mounts, effectively neutralizing the host filesystem traversal mechanism required for complex container escapes :YAMLapiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-host-path
  annotations:
    policies.kyverno.io/title: Disallow hostPath
    policies.kyverno.io/category: Pod Security Standards (Baseline)
    policies.kyverno.io/description: >-
      HostPath volumes let Pods use host directories and volumes in containers.
      Using host resources can be used to access shared data or escalate privileges
      and should not be allowed. This policy ensures no hostPath volumes are in use,
      mitigating breakout paths such as CVE-2024-21626.
spec:
  validationFailureAction: Enforce
  background: true
  rules:
    - name: host-path-check
      match:
        any:
        - resources:
            kinds:
              - Pod
      validate:
        cel:
          expressions:
            - expression: "object.spec.?volumes.orValue().all(volume, size(volume) == 0 ||!has(volume.hostPath))"
              message: "HostPath volumes are strictly forbidden to prevent container escapes."
Architectural Context: Utilizing CEL expressions directly within the validation engine significantly reduces the processing latency of the admission controller. Setting validationFailureAction to Enforce ensures that the API server natively rejects the malicious payload before the kubelet ever attempts to schedule the container on a worker node.Disrupting Lateral Movement: Network Policies and IMDS RestrictionsThe SCARLETEEL campaign relied entirely on unrestricted network access to the cloud provider's Instance Metadata Service (IMDS) at 169.254.169.254 to harvest AWS credentials and move laterally out of the Kubernetes cluster into the broader cloud account.By default, any pod scheduled on an EC2 node can route traffic to the IMDS endpoint unless explicitly blocked. Implementing strict egress controls via Container Network Interface (CNI) plugins like Calico or Cilium neutralizes this vector entirely.The following Cilium CiliumClusterwideNetworkPolicy drops all egress traffic destined for the IMDS IP, severing the attacker's path to cloud credential theft while allowing necessary host-level components (like the aws-node daemon) to function :YAMLapiVersion: cilium.io/v2
kind: CiliumClusterwideNetworkPolicy
metadata:
  name: block-imds-access
spec:
  endpointSelector:
    matchExpressions:
      # Apply to all namespaces EXCEPT critical system namespaces that may require metadata
      - key: k8s:io.kubernetes.pod.namespace
        operator: NotIn
        values:
          - kube-system
  egress:
    - toCIDRSet:
        # Define the IMDS endpoint explicitly
        - cidr: 169.254.169.254/32
    # Apply a DENY action (supported natively in newer Cilium versions or via matching lack of ALLOW)
    endpointSelector: {}
Furthermore, transitioning away from legacy Node IAM roles toward modern EKS Pod Identity or IAM Roles for Service Accounts (IRSA) ensures that workloads are cryptographically bound to specific, least-privilege roles. The EKS Pod Identity Agent intercepts metadata requests at the network layer and issues scoped temporary credentials exclusively mapped to the requesting pod's individual Service Account, rendering host-level IMDS credential harvesting obsolete and minimizing the blast radius of a compromised container.System Call Restrictions: Seccomp and AppArmorTo further restrict the execution environment within the container itself, defenders must utilize Linux kernel security modules such as Seccomp (Secure Computing Mode) and AppArmor.Threat actors frequently rely on specific system calls to escalate privileges or manipulate network configurations. For instance, preventing the CAP_NET_RAW capability prevents a compromised pod from executing ARP spoofing attacks against other workloads on the same node. By applying a strict Seccomp profile that explicitly denies unsafe system calls (e.g., ptrace, which is frequently used to inject code into other running processes), security engineers can restrict the container's functionality exclusively to what is required by the application, drastically reducing the available attack surface for exploits like CVE-2024-21626.Takeaways for the SOCThe convergence of AI-driven automation, identity exploitation, and kernel-level container escapes requires the Security Operations Center to adopt a highly technical, engineering-centric operational model. Based on the threat landscape analysis, telemetry architectures, and detection engineering principles outlined in this report, security leadership and operational teams should prioritize the following strategic mandates:Elevate the Identity Perimeter: Threat actors no longer "hack" in via traditional network exploits; they log in using stolen Service Account tokens, harvested IMDS credentials, and compromised SSH keys. The SOC must pivot from focusing solely on malware hash detection to building rigorous detections around Identity and Access Management (IAM) anomalies. Correlating Kubernetes Service Account usage across both the internal API server and external Cloud Provider APIs (e.g., AWS CloudTrail) is non-negotiable for spotting lateral movement and resource hijacking campaigns like LLMjacking.Deploy eBPF for Deep Observability: Traditional user-space EDR tooling is largely blind to container escapes, memory-resident miners, and built-in shell exfiltration techniques. The SOC must champion the deployment of eBPF-based solutions (utilizing Falco for broad alerting, and Tetragon for strict enforcement) to capture low-level system calls (execve, ptrace, openat) without the stability overhead associated with legacy kernel modules. Furthermore, security architects must carefully consider the platform restrictions of serverless compute (like AWS Fargate) against the deep visibility requirements of eBPF DaemonSets during the initial infrastructure design phase.Implement Preventative Controls Over Reactive Alerting: The unprecedented speed of modern attacks—such as the weaponization of React2Shell (CVE-2025-55182) within 30 hours of disclosure, or FAMOUS CHOLLIMA achieving administrative access in under 8 minutes—means that post-compromise alerting is often too slow to prevent critical data loss. The SOC must collaborate seamlessly with DevOps teams to deploy Admission Controllers (Kyverno, OPA Gatekeeper) and explicit Network Policies to enforce Pod Security Standards at the CI/CD pipeline. Denying hostPath mounts, restricting Linux capabilities via Seccomp, and dropping IMDS network traffic fundamentally breaks the foundational kill chains relied upon by groups like SCARLETEEL, TeamTNT, and Kinsing.Treat Detections as Software Code: Utilizing unstructured log streams directly within a SIEM leads to unmanageable false-positive rates and critical alert fatigue. Detection Engineers must formalize the creation of custom parsers to strictly normalize data into a Unified Data Model (UDM). Leveraging multi-stage query languages like YARA-L 2.0 or open-source Sigma correlations allows the SOC to chain isolated, seemingly benign events (e.g., a pod creation followed by interactive execution, culminating in an anomalous outbound network flow) into high-confidence, actionable composite alerts.By treating Kubernetes not merely as static infrastructure, but as a rich, API-driven operating system, defenders can architect visibility and control paradigms fully capable of withstanding the next generation of cloud-native adversarial operations.
